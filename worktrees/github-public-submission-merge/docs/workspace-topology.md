# Workspace Topology for Unity Generation, Training, and Release Ownership

This document maps where contributors should make changes for Unity generation/training workflows versus release packaging in this workspace.

## 1) Source-of-truth map (by concern)

| Concern | Source-of-truth repository/directory | Relative path from `core-orchestrator/` | Notes |
|---|---|---|---|
| Unity code generation + RL training logic | **External Unity MLOps module** (not currently versioned in this repo) | `../A2A_MCP/mlops_unity_pipeline.py` (expected), `../A2A_MCP/unified_bridge.py` (expected) | `core-orchestrator` currently contains no `Unity`/`mlagents` pipeline code; contributors should treat this as an external handoff boundary and validate from that module's repo root. |
| Runtime orchestration and policy wiring | `core-orchestrator` | `registry/`, `manifests/`, `scripts/`, `app/`, `docs/` | This repo is the control-plane/orchestration surface that consumes manifests/ledgers and coordinates downstream systems. |
| Knowledge-plane manifests consumed by orchestrator | SSOT workspace/repo | `ssot/` (local copy) and/or sibling path `../ssot/` in command examples | Existing dataflow docs already show orchestrator commands that consume SSOT manifests via relative path handoff. |
| Previz/render adjunct module | ADAPTCO Previz module | `adaptco-previz/` | Co-located module used as an externalized rendering/preview boundary, separate from orchestrator core runtime. |
| SSOT adjunct module | ADAPTCO SSOT module | `adaptco-ssot/` | Co-located SSOT service/module used as a dedicated data/registry boundary. |

## 2) Release artifact ownership (this repo)

The `core-orchestrator` repo is the owner for release-attestation artifacts that are committed or generated here:

| Artifact class | Owning location in this repo | Expected producer | Expected consumer |
|---|---|---|---|
| Release manifest (`manifest.json`) | `manifest.json` | Core release/update workflows in this repo | Deploy/runtime/integrity scripts in this repo and downstream operators |
| Service/runtime manifests | `manifests/` | Service authors + release automation in this repo | Orchestrator runtime + validation scripts |
| Checksums/hashes (`sha256`) | Generated by freeze/hash scripts (for example, `scripts/_freeze_common.sh`, `scripts/jcs_checksum_helper.py`) and written to output files (`.sha256`, JSON fields, or `.out/`) | Freeze/release operators in this repo | Integrity validation and replay tooling |
| Signatures/attestations | Generated/verified by signing and verification scripts (for example, `scripts/_freeze_common.sh`, `scripts/verification/verify_ndjson_ed25519.js`) | Council/release operators in this repo | Replay/verification flows before promotion |
| SBOM | **Not currently checked in at a canonical path in this repo** | Release packaging job (to be defined by pipeline owners) | Security/compliance gates before distribution |

## 3) Unity training handoff contract into `core-orchestrator`

Because Unity generation/training logic is external, contributors should hand off artifacts into `core-orchestrator` at explicit boundaries:

1. **From Unity/training repo root** (`../A2A_MCP/` expected):
   - Generated Unity asset/spec metadata (JSON/YAML).
   - Trained model artifact (for example ONNX/Barracuda output).
   - Training run metadata (algorithm, seed, step count, metrics summary).
   - Optional provenance material (hashes/signatures/SBOM from the training release process).

2. **Into this repo (`core-orchestrator/`)**:
   - Register or update routing/registry pointers in `registry/` as needed.
   - Add/update deployment/release pointers in `manifests/`.
- Seal/freeze artifacts with the local freeze scripts (e.g., `scripts/_freeze_common.sh`) and verification scripts.
   - Emit auditable outputs under `.out/`/artifact paths used by current release scripts.

3. **Validation from correct roots**:
   - Validate Unity build/training from the Unity MLOps repository root (external module).
   - Validate orchestration/manifest/signature integrity from `core-orchestrator/` root.
   - Validate SSOT manifest availability from `ssot/` or sibling `../ssot/` root, matching the command context used.
